<Info>Transaction endpoints require a **PRO API key**.</Info>

The Transactions API turns vault interactions into ready-to-sign Ethereum transaction objects. Pass the output directly to a wallet provider (ethers.js, wagmi, viem, etc.) — no ABI encoding required.

The API handles the full complexity of multi-step vault flows: ERC-20 approvals, request queues, cooldown periods, and reward claims are all exposed as ordered action arrays.

---

## Overview

### Two-step workflow

1. **Fetch context** — call `/v2/transactions/context/...` to check balances, understand the vault's current deposit/redeem state, and determine what step is next.
2. **Fetch transaction** — call `/v2/transactions/{action}/...` to get the signed transaction payload(s) for the specific action you want to execute.

### Action types

| Action | Description |
|--------|-------------|
| `deposit` | Deposit assets into the vault |
| `redeem` | Withdraw assets from the vault |
| `request-redeem` | Queue a redemption (for vaults with delayed withdrawals) |
| `request-deposit` | Queue a deposit (for vaults with delayed deposits) |
| `claim-redeem` | Claim a queued redemption after the waiting period |
| `claim-deposit` | Claim a queued deposit |
| `claim-rewards` | Claim accrued token rewards |
| `start-redeem-cooldown` | Initiate the cooldown period before redeeming |

## GET /v2/transactions/context/{userAddress}/{network}/{vaultAddress}

Returns the user's current state for a vault: balances, available deposit/redeem steps, and claimable rewards. Call this before building a transaction UI to know which action step is currently available.

```bash
GET https://api.vaults.fyi/v2/transactions/context/{userAddress}/{network}/{vaultAddress}
```

### Path parameters

<ParamField path="userAddress" type="string" required>User wallet address.</ParamField>
<ParamField path="network" type="string" required>Network name or CAIP-2 identifier.</ParamField>
<ParamField path="vaultAddress" type="string" required>Vault contract address.</ParamField>

### Response

<ResponseField name="currentDepositStep" type="string">
  The next deposit action available for this user: `deposit`, `request-deposit`, `claim-deposit`, etc.
</ResponseField>

<ResponseField name="depositSteps" type="array">
  <Expandable title="Deposit step">
    <ResponseField name="name" type="string">Step name (one of the action types above).</ResponseField>
    <ResponseField name="actions" type="array">Action names within this step.</ResponseField>
    <ResponseField name="actionsUrl" type="string">URL to fetch this step's transaction payloads.</ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="currentRedeemStep" type="string">
  The next redeem action available for this user.
</ResponseField>

<ResponseField name="redeemSteps" type="array">
  Same structure as depositSteps, for the redemption flow.
</ResponseField>

<ResponseField name="asset" type="object">
  <Expandable title="Asset with balance">
    <ResponseField name="symbol" type="string">Asset ticker.</ResponseField>
    <ResponseField name="decimals" type="integer">Token decimals.</ResponseField>
    <ResponseField name="balanceNative" type="string">Wallet balance of the deposit asset.</ResponseField>
    <ResponseField name="balanceUsd" type="string">Wallet balance in USD.</ResponseField>
    <ResponseField name="unclaimedNative" type="string">Unclaimed rewards balance in native units.</ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="lpToken" type="object">
  <Expandable title="LP token with balance">
    <ResponseField name="symbol" type="string">LP token symbol.</ResponseField>
    <ResponseField name="decimals" type="integer">LP token decimals.</ResponseField>
    <ResponseField name="balanceNative" type="string">User's current vault share balance.</ResponseField>
    <ResponseField name="balanceUsd" type="string">Vault share balance in USD.</ResponseField>
  </Expandable>
</ResponseField>

### Example

## GET /v2/transactions/{action}/{userAddress}/{network}/{vaultAddress}

Returns one or more ready-to-sign transaction objects for the requested action. Execute `actions` in order — each must confirm on-chain before the next is sent.

```bash
GET https://api.vaults.fyi/v2/transactions/{action}/{userAddress}/{network}/{vaultAddress}
```

### Path parameters

<ParamField path="action" type="string" required>
  One of: `deposit`, `redeem`, `request-redeem`, `request-deposit`, `claim-redeem`, `claim-deposit`, `claim-rewards`, `start-redeem-cooldown`.
</ParamField>
<ParamField path="userAddress" type="string" required>User wallet address.</ParamField>
<ParamField path="network" type="string" required>Network name or CAIP-2.</ParamField>
<ParamField path="vaultAddress" type="string" required>Vault contract address.</ParamField>

### Query parameters

<ParamField query="assetAddress" type="string" required>
  Address of the asset being deposited or redeemed.
</ParamField>
<ParamField query="amount" type="string">
  Amount in native token units (e.g. `1000000` for 1 USDC with 6 decimals). Required for `deposit`, `redeem`, and `request-redeem`.
</ParamField>
<ParamField query="all" type="boolean">
  For `redeem` only. When `true`, redeems the entire position. Overrides `amount`.
</ParamField>

### Response

<ResponseField name="currentActionIndex" type="number">
  Index of the action that should be executed next. Start here if continuing a previously interrupted flow.
</ResponseField>

<ResponseField name="actions" type="array">
  <Expandable title="Action">
    <ResponseField name="name" type="string">Action label (e.g. `approve`, `deposit`, `redeem`).</ResponseField>
    <ResponseField name="tx" type="object">
      <Expandable title="Transaction payload">
        <ResponseField name="to" type="string">Contract address to call.</ResponseField>
        <ResponseField name="chainId" type="integer">EVM chain ID.</ResponseField>
        <ResponseField name="data" type="string">Encoded calldata. Pass directly to the wallet.</ResponseField>
        <ResponseField name="value" type="string">ETH value to send (in wei). `null` for token-only transactions.</ResponseField>
      </Expandable>
    </ResponseField>
    <ResponseField name="simulation" type="object">
      <Expandable title="Simulation result (when available)">
        <ResponseField name="url" type="string">Link to the simulated transaction on Tenderly or equivalent.</ResponseField>
        <ResponseField name="status" type="string">`success`, `failure`, or `internal server error`.</ResponseField>
        <ResponseField name="tokensReceived" type="object">Tokens the user will receive.</ResponseField>
        <ResponseField name="tokensSpent" type="object">Tokens the user will spend.</ResponseField>
      </Expandable>
    </ResponseField>
  </Expandable>
</ResponseField>

### Example — deposit 1000 USDC

```bash
curl -H "x-api-key: YOUR_API_KEY" \
  "https://api.vaults.fyi/v2/transactions/deposit/0xYourWallet/mainnet/0xVaultAddress\
```

### Executing actions with ethers.js

```js
import { ethers } from 'ethers';

const provider = new ethers.BrowserProvider(window.ethereum);
const signer = await provider.getSigner();

for (const action of actions) {
  const tx = await signer.sendTransaction({
    to:    action.tx.to,
    data:  action.tx.data,
    value: action.tx.value ?? 0n,
  });
  await tx.wait();
}
```

## GET /v2/transactions/rewards/context/{userAddress}

Returns all claimable reward balances for a user across every vault and network.

```bash
GET https://api.vaults.fyi/v2/transactions/rewards/context/{userAddress}
```

### Path parameters

<ParamField path="userAddress" type="string" required>User wallet address.</ParamField>

### Response

<ResponseField name="claimable" type="object">
  Rewards organized by network name. Each network contains an array of claimable reward objects:
  <Expandable title="Claimable reward">
    <ResponseField name="claimId" type="string">Unique identifier for this reward — pass to the claim endpoint.</ResponseField>
    <ResponseField name="asset" type="object">
      <Expandable title="Reward asset">
        <ResponseField name="symbol" type="string">Reward token symbol.</ResponseField>
        <ResponseField name="claimableAmount" type="string">Amount claimable in native token units.</ResponseField>
        <ResponseField name="claimableAmountInUsd" type="string">USD value of claimable amount.</ResponseField>
      </Expandable>
    </ResponseField>
    <ResponseField name="sources" type="array">Protocols where these rewards originated.</ResponseField>
    <ResponseField name="actionUrl" type="string">Direct URL to claim this specific reward via the transactions API.</ResponseField>
  </Expandable>
</ResponseField>

### Example

```bash
curl -H "x-api-key: YOUR_API_KEY" \
  "https://api.vaults.fyi/v2/transactions/rewards/context/0xYourWallet"
```

```json
{
  "claimable": {
    "mainnet": [
      {
        "claimId": "claim-0xABC-mainnet-MORPHO",
        "asset": {
          "symbol": "MORPHO",
          "claimableAmount": "142000000000000000000",
          "claimableAmountInUsd": "89.50"
        },
        "sources": [{ "protocol": { "name": "Morpho" } }],
        "actionUrl": "https://api.vaults.fyi/v2/transactions/rewards/claim/0xYourWallet?claimIds=..."
      }
    ]
  }
}
```

## GET /v2/transactions/rewards/claim/{userAddress}

Generates transaction payloads to claim one or more rewards. Pass the `claimId` values from the context endpoint.

```bash
GET https://api.vaults.fyi/v2/transactions/rewards/claim/{userAddress}
```

### Path parameters

<ParamField path="userAddress" type="string" required>User wallet address.</ParamField>

### Query parameters

<ParamField query="claimIds" type="array" required>
  Array of `claimId` values from `/v2/transactions/rewards/context/...`. Rewards are batched by network — pass all IDs from the same network in a single call.
</ParamField>

### Response

Response is keyed by network name. Each network entry contains `currentActionIndex` and `actions` with the same shape as the main transactions endpoint.

```json
{
  "mainnet": {
    "currentActionIndex": 0,
    "actions": [
      {
        "name": "claim",
        "tx": {
          "to": "0xClaimContract",
          "chainId": 1,
          "data": "0x...",
          "value": null
        }
      }
    ]
  }
}
```
